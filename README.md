# AI Health & Fitness Planner

Streamlit app that builds **diet** and **fitness** plans with LangChain, then answers follow-up questions using your profile and both plans. You can use **Ollama** (local) or **Google Gemini** (API).

This project is informational only—not medical or professional advice.

## Requirements

- Python 3.10+ (tested in development with newer 3.x; LangChain may log a Pydantic warning on very new Python versions)
- For **Ollama**: [Ollama](https://ollama.com/) installed and running
- For **Gemini**: a Google AI Studio / Gemini API key

## Setup

From the project root (`AIHealthAssistant/`):

### 1. Virtual environment

```bash
python3 -m venv .venv
source .venv/bin/activate
```

On Windows: `.venv\Scripts\activate`

### 2. Dependencies

```bash
pip install --upgrade pip
pip install -r requirements.txt
```

To refresh to newer releases later:

```bash
pip install -U -r requirements.txt
```

### 3. Environment variables

Copy the example file and edit as needed:

```bash
cp .env.example .env
```

| Variable | When needed | Purpose |
|----------|-------------|---------|
| `GOOGLE_API_KEY` | Only if you choose **gemini** in the app | Gemini API authentication |
| `OLLAMA_BASE_URL` | Optional | Ollama HTTP API (default `http://localhost:11434`) |
| `DEFAULT_LLM_PROVIDER` | Optional | `ollama` or `gemini` for the initial sidebar selection |
| `DEFAULT_OLLAMA_MODEL` | Optional | Default Ollama model name in the sidebar |
| `DEFAULT_GEMINI_MODEL` | Optional | Default Gemini model name in the sidebar |

You can run with **Ollama only** and leave `GOOGLE_API_KEY` empty until you switch the sidebar to Gemini.

## Ollama (local)

1. Start the Ollama service (varies by OS; often it runs automatically after install).
2. Pull a chat model, for example:

   ```bash
   ollama pull llama3.2
   ```

3. In the app sidebar, choose provider **ollama** and set the model name to match (e.g. `llama3.2`).
4. Use **Test connection** in the sidebar to confirm the model responds.

## Google Gemini

1. Create an API key in [Google AI Studio](https://aistudio.google.com/) (or your Google Cloud Gemini setup).
2. Set `GOOGLE_API_KEY` in `.env` (or export it in your shell).
3. In the app sidebar, choose provider **gemini** and pick a model your key supports (default in `.env.example` is a reasonable starting point).

## Run the app

From the **workspace root** (same directory as `app.py`):

```bash
source .venv/bin/activate
streamlit run app.py
```

Open the URL Streamlit prints (usually `http://localhost:8501`).

**Smoke check (headless, short run):**

```bash
source .venv/bin/activate
streamlit run app.py --server.headless true --browser.gatherUsageStats false
```

Stop with Ctrl+C. You should see “You can now view your Streamlit app in your browser.”

Configure at least one backend (Ollama with a pulled model, or Gemini with a key) before generating plans.

## GitHub Actions (CI and auto versioning)

The workflow [.github/workflows/ci.yml](/home/lucky/Projects/AIHealthAssistant/.github/workflows/ci.yml) runs on **pushes** and **pull requests** to `main` or `master`:

1. **Test** — Python **3.11** and **3.12**: `pip install -r requirements.txt`, `python -m compileall`, then import checks for `config`, `chains`, `llm_factory`, `prompts`, `version`, and `app`.
2. **Auto version tag** — On **push** to `main` or `master` only (after all matrix jobs succeed), a job bumps the **patch** semver and pushes a new **git tag** (e.g. `v0.1.2`) using [anothrNick/github-tag-action](https://github.com/anothrNick/github-tag-action). If the repo has no prior `v*` tags, it starts from `0.1.0` and bumps from there.

Tags are the automatic per-push version identity in CI. [`version.py`](/home/lucky/Projects/AIHealthAssistant/version.py) `__version__` is **not** rewritten by the workflow; align it manually with tags if you want a single displayed number in code.

## Q&A context and long plans

The MVP passes the **full** user profile, **full** diet plan, and **full** fitness plan into every follow-up question, plus a short **chat history** string. For very long plans, you may eventually hit model context limits; mitigations (not implemented yet) include summarizing plans, truncating with an explicit “[truncated]” note, or retrieving only relevant sections. Until then, keep plans reasonable in length or switch to a larger-context model.
